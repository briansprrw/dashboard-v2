## 1. Launch definition (V2)

Dash2 V2 launches as an **authenticated-only, single responsive web application** for users **migrated from V1**. It has **no public output, no onboarding, and one dashboard per user** at launch. The defining deliverable is a high-density, glanceable, responsive task dashboard.

### In launch scope

- Google sign-in for migrated users; 30-day sliding, immediately-revocable sessions.
- One dashboard per user; automatic 1–3 column responsive flow through 1920×1080; desktop, phone, tablet, and smart-frame-**sized** rendering.
- Glance and Standard presentation modes; Manage mode for management surfaces.
- Task CRUD, move, quick-complete; fixed launch sort; status/priority/due/notes/emoji-flags; private tasks and private notes.
- Viewer / Editor / Owner / Admin roles enforced server-side.
- Recycle-bin lifecycle (30 days) for tasks, Lists, and accounts, with restore and bounded early purge.
- Owner-only task-history field values; separate allowlisted administrative audit/recovery metadata.
- Per-device display preferences (mode, 7-step zoom, density, 4 themes, per-device emoji, clock/date, refresh 60/10/600s, closed-task visibility, validated due-band thresholds, and 1–3-column min/max bounds).
- Offline-visible/read-only behavior with `Offline` label.
- Site-wide `noindex`.
- Migration of V1 users/roles, Lists/ownership, and all feasible task data, with reconciliation.

### Explicit non-goals / deferred to V2.1 (not launch)

Deferred (architectural seams may exist, but no launch UX): **public dashboards, public usernames, public field projection, public preview, dedicated smart-frame display sessions, synchronized display profiles, multiple saved dashboards, dashboard sharing, invite-code onboarding / new-account creation, search, filters, manual/selectable sort, external/developer API, additional identity providers, cryptographic/owner-key protection against direct database or infrastructure access, ultrawide & portrait-desktop layouts, bulk actions, recurring tasks, tags, subtasks, attachments, sheet templates, and enhanced Cloudflare edge-abuse/DDoS hardening beyond the V2 launch baseline.** Search indexing is not deferred: it is permanently prohibited. Application-level Admin denial for private content is V2 launch scope, not deferred.

Notes are **never** public. Private tasks are excluded from any future public output by default.

Administrative authority and protected-content visibility are separate policies: V2 Admin cannot read private tasks, private notes, or task-history field values.

## 3. Lifecycle contracts (launch)

**Tasks:** Editor recycles → Owner or Admin restores or permanently deletes. Admin recovery uses opaque identity and allowlisted metadata without revealing protected task content or task-history field values. Recycle/restore preserves task history; permanent delete purges task + history. 30-day auto-purge eligibility.

**Lists:** Windows folder semantics — List + all contained tasks/history move and restore as one unit. Owner/Admin restore; Owner/Admin early permanent-delete (cascades to contained tasks/history) with confirmation. 30-day window.

**Accounts:** Admin-controlled recycle (no self-service delete). Recycle disables auth and soft-deletes the account + every owned List/task as a unit; the account's Viewer/Editor memberships are suspended; owned Lists disappear for other members until restore. One admin action restores the whole unit; one confirmed admin action permanently deletes it early.

**Task history:** actor, time, action, changed fields, full before/after (incl. names/notes). Field values are List-owner-only. Admin may receive only allowlisted administrative recovery/audit metadata and cannot read before/after values or other protected history fields. Separate authorization from ordinary reads. Survives recycle/restore; purged with permanent delete.

**Administrative audit:** separate append-only stream for role/ownership/membership changes, account/List recycle/restore, permanent purge, (future) public config, and admin overrides. Metadata allowlisted; no private content, credentials, tokens, or session IDs.

## 6. Isolation & V1-preservation rules (restated as contract)

- Separate repo, Worker, D1, KV, OAuth callback, hostname, deployment path. Runtime code has **no** V1 binding.
- V1 stays live and unchanged as source of truth until M8 is approved; V1 is a **read-only** migration source only.
- Public output at launch: none. Site-wide `noindex`. Future public output uses an allowlist projection.
- Every sheet/List has exactly one owner at all times.

## 7. Resolved detail decisions

The former M0.2 open details are resolved: OD-1 uses `recycled_at` plus explicit `recycled` account state (M0-D22); OD-2 uses seven due bands with locally configurable validated thresholds (M0-D23); and OD-3 uses automatic columns within device-local min/max bounds, with a provisional safe-width minimum fallback that M3 must expose and evaluate rather than assume is permanently correct (M0-D24).
