## Safe evidence strategy

M0 is a decisions milestone with **no application code and no live-system access in this worktree**:

- There is no running Dash2 app to screenshot (M1 builds it).
- The V1 reference screenshot and the live V1 database contain **real task names, notes, and emails**, which CLAUDE.md and AR §10 forbid placing in M0 evidence.
- Per M0-D25, M0 does **not** query or profile V1. Any later source profiling requires separate authorization in M6.

The safe M0.4 deliverable is: (1) this **reference-capture specification and index**, plus sanitized screenshots of the coded mockup at VP-1..VP-9 labeled “mockup, not the app”; (2) a **sanitized V1 behavior inventory** based only on existing audits; (3) a **non-live source-shape inventory** based only on schema/audit evidence; and (4) a **synthetic migration-fixture specification**. M3/M7 replace the interim mockup captures with application evidence.

### What each capture must show (per viewport, where applicable)

- Glance mode with the full task grammar `[status][name][note][emoji][due][priority]`.
- All seven due-color bands present (overdue / today / soon / soonish / future / complete / unscheduled) — see OD-2; working baseline is 7 bands.
- At least one long task name (truncation behavior) and one `TBD`/unscheduled task.
- Collapsed and expanded section states.
- The narrow task editor/bottom-sheet reference is captured in the focused VP-5 asset.
- The coded mockup exposes only its current dark presentation and no reduced-motion or zoom controls. Four-theme, reduced-motion, max-zoom, wide-dialog, and management-state proof therefore belongs to the M3/M7 **application** evidence set; no mockup variant is fabricated as proof.

### Sanitization rules for every captured asset

- Use **synthetic** task/List/user data only (the fixture in §4). Never the reference screenshot's real content, real emails, or production notes.
- No email addresses, real names, session cookies, or URLs containing tokens in-frame.
- Store under `docs/mockups/reference/` (or a milestone evidence path) with descriptive, non-sensitive filenames (e.g. `vp1-narrow-glance-dark.jpg`).
- The existing **coded mockup** ([`docs/mockups/dash2.html`](../mockups/dash2.html)) is the interim visual baseline and uses synthetic data. These assets are explicitly labeled as mockup evidence, not application proof.

## 2. Sanitized V1 behavior inventory (reference contracts to preserve or intentionally drop)

Built from the sanitized audits ([Codex](../audits/2026-07-22-codex-review.md), [Fable](../audits/2026-07-22-claude-fable-review.md), [rewrite-vs-revamp](../audits/2026-07-22-rewrite-vs-revamp.md)). These are **migration/reference facts**, not V2 design authority. No V1 content was read for this; only the audits.

| V1 behavior | Source | V2 disposition |
|---|---|---|
| Google OAuth + one-time KV state nonce; server-side KV sessions; per-request role/allowlist revalidation | CX positive, FB §4 | **Preserve the pattern** (M2). Strong; keep. |
| `access_control.can_see` = single permission bit (read == write) | CX C2, FB F2 | **Replace** with Viewer/Editor/Owner memberships (M0-D3). Do not migrate. |
| Public endpoint `SELECT t.*` (notes, IDs, timestamps, email-derived slug exposed) | CX C1, FB F1 | **Do not reproduce.** No public output at launch; future public uses allowlist projection (M0-D6/D7). |
| Sheet slugs derived from email (`email.replace('@','_')`) | FB F1/F13 | **Drop.** V2 uses UUIDs; no email-derived identifiers (M0-D12). |
| `MAX(row_index)+1` ordering with race risk | CX §DB, FB F12 | **Replace** with `sort_key` spacing + deterministic tie-break (TA). |
| Status set: not started / in progress / pending / blocked / complete / cancelled | Mockup (mirrors V1 `CONFIG`), PP B6 | **Adopt** as launch status enum. |
| Priority set: low / medium / high / urgent | Mockup, PP B7 | **Adopt** as launch priority enum. |
| Settings string-typed, reach HTML/CSS sinks (self-scoped) | CX C7, FB hyp-6 | **Do not migrate settings** (M0-D13); V2 uses validated preference schema + themes. |
| Invite-code seat provisioning not atomic | CX C3, FB F11 | **Not migrated / no launch onboarding** (M0-D13). Invite seam may exist for V2.1 with atomic redemption. |
| Migrations 0001–0006; no FKs/CHECKs; app-enforced invariants | CX C8, FB F6 | V2 starts fresh with FKs/CHECKs where D1 permits (M2). V1 schema is a migration source only. |
| Desktop/mobile duplicated render paths | CX C10, FB F8 | **Do not reproduce.** One responsive component/action model (M0-D8/D11). |
| README documents obsolete GAS/PIN architecture | CX C12, FB F9 | V1-repo doc debt; not a Dash2 artifact. Out of M0 scope. |

**V1 behavior contracts worth an explicit migration test (M6):** OAuth/session semantics equivalence is *not* required (greenfield auth); what must be preserved is **task data fidelity** (names, notes, dates, statuses, priorities, flags, ordering, valid timestamps) and **ownership**. These become AC-X2 / AC-X4 checks.

## 3. Non-live source-data shape inventory

This is the expected **shape** of the V1 source for migration design, expressed as structure and known-anomaly categories — **no query, no live counts, no content**. All facts derive from existing sanitized audits and the approved mapping, not from reading V1. If Brian separately authorizes source profiling during M6, M6 creates fresh evidence under its own controls.

### Entities and the fields that must survive migration

| V1 entity (approx.) | Fields to migrate | V2 destination | Known-anomaly categories to expect |
|---|---|---|---|
| Users / `allowed_users` | identity email (→ identity record), global role, state | `users` + `user_identities` | duplicate/case-variant emails; disabled users; admin vs user role |
| `sheets` | display name, owner email | `sheets` (`owner_user_id`, `legacy_source_id`) | email-derived slug; owner email not matching a user; orphan sheets |
| `tasks` | name, notes, due_date, status, priority, emoji flags, ordering, created/updated/closed timestamps | `tasks` | null/invalid due dates; unknown status/priority casing; missing timestamps; tasks referencing a missing sheet; very long names/notes |
| `access_control` | — | **not migrated** | dropped by decision; future M6 reconciliation accounts for it only if source access is authorized |
| `settings` | — | **not migrated** | dropped by decision; future M6 reconciliation accounts for it only if source access is authorized |
| `invite_codes` | — | **not migrated** | dropped by decision; future M6 reconciliation accounts for it only if source access is authorized |

### Boundary for later M6 work

M0 records expected entities and anomaly categories only. It does not establish counts, ranges, user-state totals, source-row dispositions, or any other live-source baseline. Those are M6 migration/reconciliation concerns only if Brian separately authorizes read-only V1 access.

## 4. Synthetic migration-fixture specification

A **synthetic** fixture (invented values, no production content) that exercises every normal and edge case, for M2 schema tests and M6 rehearsals. Implemented in M1/M6 as fixture files; specified here.

### Users (synthetic)

| Case | Spec |
|---|---|
| U-normal | active user, role `user`, unique lowercase email |
| U-admin | active user, role `admin` |
| U-mixedcase | email with uppercase → must normalize; must not create a duplicate |
| U-disabled | pre-disabled state → cannot authenticate after migration |
| U-owner-multi | owns several Lists (cascade/ownership test) |

### Lists (synthetic)

| Case | Spec |
|---|---|
| L-normal | owned by U-normal, plain display name |
| L-legacy-slug | source slug looks email-derived → must NOT surface in V2 IDs |
| L-owner-missing | owner reference that doesn't resolve → **quarantine**, not invent an owner |
| L-empty | zero tasks |
| L-many | large task count (density/perf) |

### Tasks (synthetic — no real content)

| Case | Spec |
|---|---|
| T-each-status | one per status: not started / in progress / pending / blocked / complete / cancelled |
| T-each-priority | one per priority: low / medium / high / urgent |
| T-due-bands | one per due band: overdue / today / soon / soonish / future / unscheduled |
| T-undated | `due_date` null → TBD |
| T-invalid-date | malformed source date → rejected/quarantined with reason |
| T-long-name | name at/над the max length cap (truncation + validation) |
| T-long-notes | notes at the max length cap |
| T-private | private task (owner-visible; hidden from Viewer/Editor/Admin read paths) |
| T-private-note | ordinary task with a private note (note owner-visible; hidden from Viewer/Editor/Admin) |
| T-admin-private-recovery | Admin restores/purges a private task by opaque ID without receiving task fields, note content, or history field values |
| T-emoji-flags | bounded emoji-flag list |
| T-orphan-sheet | references a missing List → quarantine |
| T-closed | has a valid `closed_at`; complete + cancelled variants (separate visibility) |
| T-recycled | in the recycle bin with a recycle timestamp (30-day lifecycle) |

### Excluded-by-decision fixtures (present in source, must be dropped and only counted)

- Shares / `access_control` rows.
- Settings rows.
- Invite codes.

### Fixture invariants the migration must prove (M6)

- No migrated List is ownerless (AC-X6).
- No task/membership references a missing user/List (AC-X6).
- Two consecutive imports are logically identical (AC-X5).
- Every source row accounted for, including intentional drops (AC-X4).
- Fixture emails/names/notes are entirely invented and contain no real/private content. Migration logs and shareable evidence do not print fixture values or credentials (AC-X8).
